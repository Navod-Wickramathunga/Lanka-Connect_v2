# Lanka Connect – Project Report Sections

---

## 1. Abstract

The rapid growth of mobile internet usage in Sri Lanka has created significant opportunities for digital platforms that connect service consumers with local service professionals. However, despite this growth, the local service industry continues to operate largely through informal, fragmented channels that do not leverage digital technology effectively. Lanka Connect was designed and developed as a direct response to this gap — a comprehensive, cross-platform service marketplace application that brings together service seekers and service providers in Sri Lanka through a structured, technology-driven platform that handles the full lifecycle of a service transaction from discovery to payment and review.

The platform was built as both a native-capable mobile application (supporting Android and iOS) and a responsive web application, using Flutter as the unified front-end framework. This single-codebase approach allowed the development team to deliver consistent user experiences across multiple platforms simultaneously, significantly reducing development overhead. Firebase was selected as the serverless back-end infrastructure, providing a fully managed suite of cloud services that eliminated the need to provision or manage servers, enabling rapid development and deployment while ensuring production-grade reliability.

The mobile application serves as the primary interface for two distinct user groups: service seekers and service providers. Service seekers can browse, search, and filter approved service listings across nine predefined categories — Cleaning, Plumbing, Electrical, Carpentry, Painting, Gardening, Moving, Beauty, and Tutoring — using a rich set of discovery tools including keyword search, category and location filters, price range selection, and a geolocation-based "Near Me" feature powered by Google Maps. Once a suitable service is identified, seekers can initiate a booking, communicate directly with the service provider through a real-time in-app chat system, process payments through multiple methods, and submit a star rating and review upon service completion. Service providers, operating through the same mobile application, can create and manage service listings, respond to incoming booking requests, mark services as completed, track their earnings and performance through a dedicated provider dashboard, and manage their bank account details for payment collection.

The web application functions as a dedicated administration panel restricted to platform administrators. Through this dashboard, administrators can moderate newly submitted service listings by approving or rejecting them through a structured queue with Service Level Agreement (SLA) tracking indicators, manage promotional banners and marketing campaigns, create and manage discount offers for specific services or providers, oversee payment transactions, and monitor platform-wide analytics and system metrics.

The back-end architecture employs Cloud Firestore as a real-time NoSQL document database that synchronizes data across all connected clients instantly without requiring manual refresh cycles. Firebase Authentication handles secure user identity management through email and password credentials with support for password recovery via email link. Firebase Cloud Storage hosts all media assets including service listing images, user profile photographs, and banner graphics. Firebase Cloud Messaging delivers push notifications to mobile devices for events such as booking status changes, service approvals, and payment confirmations. Ten Cloud Functions, implemented in TypeScript on a Node.js 22 runtime and deployed on Firebase Cloud Functions v2, handle critical server-side business logic including the automated enforcement of service listing statuses, atomic provider rating aggregation, payment gateway integration through the PayHere API, bank transfer verification, and automated receipt dispatch through the Twilio SMS and SendGrid email communication services.

The system implements a comprehensive role-based access control model across four user roles — Seeker, Provider, Admin, and Guest — enforced through Firestore security rules that validate permissions at the document level on every read and write operation. The platform manages data across thirteen Firestore collections encompassing user profiles, service listings, bookings, service requests, reviews, chat messages, notifications, payment records, payment receipts, provider bank accounts, saved payment methods, promotional banners, marketing promotions, and discount offers. Lanka Connect has been deployed to a live production environment, with the web administration dashboard hosted on Firebase Hosting and the mobile application prepared for distribution through the Google Play Store and Apple App Store. A multi-environment strategy supporting production, staging, and local Firebase emulator configurations supports ongoing development, testing, and quality assurance without impacting the live system.

## 2. Problem Identification

The development of Lanka Connect was motivated by a careful analysis of the structural deficiencies in the Sri Lankan local service industry. While mobile internet penetration in Sri Lanka has grown substantially in recent years, the process by which individuals find and engage local service professionals — plumbers, electricians, cleaners, carpenters, tutors, and others — remains largely informal, unstructured, and inefficient. The following problems were identified as the primary justification for developing a dedicated technology platform to address them:

**Problem 1 – Lack of a Centralized Service Discovery Platform:**

In Sri Lanka, individuals seeking household and professional services such as plumbing, electrical work, cleaning, carpentry, tutoring, and gardening rely predominantly on word-of-mouth referrals, printed classified advertisements, community Facebook groups, and informal neighbourhood networks. There is no centralized, technology-driven marketplace that aggregates service providers across multiple service categories and geographical locations into a single searchable and filterable platform that is accessible from a mobile device or web browser.

This fragmentation creates a significant discoverability problem. A customer in need of an electrician in Kandy, for example, has no reliable digital channel to browse available electricians, compare their pricing, check their ratings, or view their prior work. The search process is unpredictable, time-consuming, and highly dependent on personal networks, meaning that many qualified service providers remain invisible to potential customers simply because of the absence of a discovery mechanism. Geographical barriers further compound this problem, as providers who operate slightly outside a seeker's immediate social circle are effectively excluded from consideration regardless of their quality or competitiveness. This problem directly motivated the creation of a centralized, multi-category service marketplace as the core purpose of Lanka Connect.

**Problem 2 – Absence of Trust and Quality Assurance Mechanisms:**

Without a formal digital platform that records, aggregates, and publicly displays service quality indicators, service seekers have no reliable way to evaluate the competence, reliability, or professionalism of a service provider before engaging them. In the informal market, provider reputation exists only in private word-of-mouth, which is not scalable, is not verifiable, and is highly susceptible to bias. There is no standardized system for collecting structured customer feedback, displaying aggregate ratings, or tracking individual provider performance over time.

This absence of transparency creates a market for lemons problem — seekers who cannot distinguish high-quality providers from low-quality ones are rationally cautious about engaging unknown providers, which suppresses market activity and deprives skilled providers of clients they would otherwise attract. From the provider perspective, the lack of a formal accountability mechanism reduces the incentive for continuous quality improvement, since excellent service is not rewarded with visible, quantifiable recognition. Furthermore, without administrative oversight of who is permitted to list services on a platform, there is no barrier to fraudulent or unqualified actors misrepresenting their capabilities, creating safety and financial risks for seekers.

**Problem 3 – Inefficient Booking and Communication Processes:**

The traditional approach to engaging a service provider involves a series of unstructured interactions: a phone call to determine availability, verbal negotiation of a price, manual scheduling, and follow-up calls if the appointment needs to change. None of these interactions are recorded, verifiable, or transparent to a third party. There is no structured workflow that assigns clear responsibilities, tracks the progression of a service engagement, or creates an immutable record of what was agreed between the parties.

When disputes arise — over pricing, the scope of work performed, whether the service was completed satisfactorily, or whether payment was made — there is no platform-provided evidence to refer to. Communication happens through personal phone numbers or WhatsApp messages, which are not linked to a formal transaction record. The absence of a structured booking management system with defined state transitions means that both seekers and providers lack visibility into the current status of their engagements, leading to unnecessary follow-up communication, misunderstandings, and missed appointments. This problem becomes especially pronounced when a seeker wants to manage multiple concurrent service bookings, which is essentially impossible through informal channels.

**Problem 4 – Insecure and Limited Payment Options:**

The overwhelming majority of local service transactions in Sri Lanka are settled through cash, either at the time of service delivery or in advance as a deposit. Cash transactions are inherently insecure for both parties: seekers face the risk of making advance payments without receipts or recourse if the service is not delivered as agreed, while providers face the risk of non-payment after completing the work. Cash also provides no transaction record, making it impossible to generate financial summaries, issue formal receipts, or provide evidence of payment if a dispute arises.

There is no existing integrated digital payment mechanism for local services that supports the full spectrum of payment preferences — credit and debit card processing, bank transfers, and saved payment methods — while simultaneously maintaining a complete audit trail of all transactions, generating and delivering formal payment receipts, and providing a platform administrator with oversight capability. The absence of digital payment infrastructure not only limits the formal economic activity captured on the platform but also creates vulnerability to payment-related disputes that damage trust between seekers and providers.

**Problem 5 – Lack of Administrative Oversight and Service Quality Control:**

Even if a digital platform for service discovery were to be created without a governance layer, it would quickly become populated with unverified, misleading, or fraudulent service listings that would undermine user trust and damage the platform's reputation. In the absence of a moderating authority, any person could list any service at any price with any description, and seekers would have no basis for trusting the information presented. There is a fundamental need for an administrative oversight mechanism that reviews and approves service listings before they become publicly visible, monitors platform activity, manages promotional and commercial aspects of the marketplace, and retains the authority to remove non-compliant listings.

Without such a governance layer, the platform would be unable to enforce consistent quality standards, respond to user complaints, or manage commercial relationships with partner businesses. The absence of administrative tools also limits the platform's ability to run targeted promotions, offer discounts to incentivize user acquisition, and conduct ongoing platform management activities necessary for the long-term health of the marketplace.

## 3. Objectives

Based on the five identified problems, the following six objectives were defined to guide the design, development, and deployment of the Lanka Connect platform. Each objective corresponds directly to one or more of the identified problems and specifies the technical and functional outcomes that the system must achieve.

**Objective 1 – Develop a Centralized Multi-Category Service Marketplace:**
Design and implement a cross-platform application encompassing both a mobile application (Android and iOS) and a web application that aggregates service providers across nine predefined service categories — Cleaning, Plumbing, Electrical, Carpentry, Painting, Gardening, Moving, Beauty, and Tutoring — into a single, publicly accessible marketplace. The marketplace must enable service seekers to discover relevant services through multiple discovery mechanisms: keyword search by service title and description, category-based filtering, location-based filtering by district and city, price range filtering, and geolocation-based proximity discovery using the device's GPS and Google Maps integration to show services within a configurable radius of the user's current location. Service listing cards must present sufficient information for informed browsing, and a full service detail screen must provide comprehensive information including images, description, pricing, provider profile, aggregate rating, and an embedded map view displaying the service location.

**Objective 2 – Implement a Trust and Quality Assurance Framework:**
Develop a comprehensive, server-side-enforced trust and quality framework comprising two components. First, a review and ratings system that allows only seekers who have completed a booking to submit structured feedback (star rating 1–5 and a textual review), with the provider's average rating and review count automatically updated through atomic server-side computation to prevent data inconsistencies. Second, an administrative service moderation workflow where all newly submitted service listings are automatically set to a "pending" status by a server-side Cloud Function — preventing any possibility of provider self-approval — and remain hidden from seekers until explicitly approved by a platform administrator through the dedicated web administration dashboard. Upon approval, the provider must receive an immediate push notification confirming the status change.

**Objective 3 – Build a Structured Booking and Real-Time Communication System:**
Create a complete booking lifecycle management system with formally defined state transitions — Pending → Accepted or Rejected → Completed or Cancelled — where each transition is validated both at the application layer and enforced at the database security rule layer, ensuring that invalid state changes are rejected regardless of how the request is made. The booking system must support two initiation models: direct booking by a seeker on an approved service, and a request-based model where seekers post service requests that providers can respond to. Additionally, integrate a real-time in-app chat system — backed by Cloud Firestore live listeners — that links each conversation to a specific booking, restricts access to only the two participants (seeker and provider), and eliminates the need for external messaging applications during the service engagement process.

**Objective 4 – Integrate Secure Digital Payment Processing:**
Implement a multi-method digital payment system within the mobile application that supports: (1) credit and debit card payments processed through the PayHere payment gateway, a licensed Sri Lankan payment service provider, including webhook-based callback verification to confirm payment outcomes; (2) direct bank transfer payments with a structured submission and administrative verification workflow; and (3) saved card tokenization allowing returning users to complete payments without re-entering card details. The system must track payment statuses through a defined state machine, apply eligible discounts and promotional offers at checkout, and automatically generate and deliver formal payment receipts via SMS through Twilio and email through SendGrid upon successful payment confirmation.

**Objective 5 – Establish an Administrative Control and Monitoring Dashboard:**
Develop and deploy a web-based administration panel, accessible exclusively to authenticated administrator accounts, that provides comprehensive tools for governing the Lanka Connect marketplace. This includes a service moderation queue with search, status-based tabs, SLA tracking indicators, and single-action approve/reject capabilities with immediate push notification to the affected provider; banner management allowing the creation and activation management of promotional banners displayed to mobile users; marketing promotion management for time-bound campaigns; discount offer creation supporting both flat-amount and percentage-based discounts with targeting by service, provider, or category; user account management including role modification; and an analytics dashboard displaying system-wide metrics and performance visuals through chart components.

**Objective 6 – Implement a Secure, Scalable, and Real-Time Back-End Architecture:**
Design and deploy a serverless back-end architecture utilizing Firebase's managed cloud services — Cloud Firestore for real-time document storage, Firebase Authentication for identity management, Firebase Cloud Storage for media asset hosting, Cloud Functions v2 for event-driven and callable server-side logic, and Firebase Cloud Messaging for push notification delivery — that provides real-time data synchronization without explicit polling, scales automatically with user demand, and requires no infrastructure management. The architecture must implement comprehensive role-based access control through Firestore security rules that enforce least-privilege access across all thirteen data collections, validate cross-document consistency on every write operation, and prevent any unauthorized data manipulation by enforcing business rules at the database layer. The deployment must support a multi-environment strategy with separate Firebase projects for production, staging, and local emulator use, enabling continuous development and testing cycles without disrupting the live production system.

## 4. Conclusion

### 4.1 Achievement of Objective 1 – Centralized Multi-Category Service Marketplace

Objective 1 was fully achieved through the design, development, and production deployment of the Lanka Connect mobile application, which now serves as a live, operational service marketplace accessible to service seekers and providers across Sri Lanka. The application aggregates service listings across all nine predefined categories — Cleaning, Plumbing, Electrical, Carpentry, Painting, Gardening, Moving, Beauty, and Tutoring — providing a single platform that eliminates the need for seekers to consult multiple fragmented channels when looking for a service professional.

The service discovery system was built around a multi-layered filtering architecture. The primary service list screen retrieves data from the Cloud Firestore `services` collection with a server-side query filtering only listings with `status == 'approved'`, ensuring that only moderated, administrator-approved services appear to seekers. From there, the client-side `ServiceDiscoveryFilter` model encapsulates the user's current filter state — category, district, city, price range, search keyword, and proximity radius — allowing the UI to apply multiple independent filters simultaneously without requiring separate database queries for each combination. The keyword search scans service titles and descriptions in real time, the category filter presents all nine service types as a horizontal scrollable category bar, and the location filter allows district and city selection from predefined picklists.

The geolocation-based "Near Me" discovery feature was implemented using the `geolocator` Flutter package to acquire the device's current GPS coordinates, combined with a custom `GeoUtils` utility that calculates the Haversine-formula distance between the user's location and the stored latitude and longitude coordinates of each service. Users can configure the search radius from 10 to 100 kilometres, and the list dynamically filters to show only services within the chosen distance. The Google Maps Flutter plugin renders a dedicated service map view displaying all approved services as interactive markers pinned to their registered coordinates, allowing seekers to visually assess the geographical distribution of services and tap any marker to preview service details before navigating to the full detail screen.

Service providers use the mobile application to create listings through a comprehensive multi-field creation form that captures the service title, category selection from the nine predefined options, pricing, a descriptive text body, GPS-acquired coordinates with a map confirmation preview, and multiple images uploaded to Firebase Cloud Storage under path-based access control rules. The provider's service management screen presents their own listings organized by approval status, allowing them to edit or delete their listings at any time. The production environment currently serves live service listings and is accessible to authenticated users across Android and iOS devices.

### 4.2 Achievement of Objective 2 – Trust and Quality Assurance Framework

Objective 2 was fully achieved through the implementation of a dual-layer quality assurance system, addressing both the supply-side quality (service listing verification) and the demand-side quality signalling (ratings and reviews) that were identified as absent in the informal service market.

The service moderation workflow was designed with a fundamental security principle: no service provider should ever be able to make their own listing publicly visible without independent administrative review. To enforce this, the `setServicePendingOnCreate` Cloud Function — deployed as a Firestore `onCreate` document trigger on the `services` collection — immediately overwrites the status field of any newly created service document with the value `'pending'`, regardless of whatever status value the client application may have submitted. This server-side enforcement is tamper-proof because it executes in the Firebase back end using the Firebase Admin SDK, which operates with full administrative privileges that bypass all client-facing security rules. The result is that the `status: 'pending'` assignment is guaranteed on every service creation event, without dependency on client-side cooperation.

The admin web dashboard's service moderation screen presents all pending services in a dedicated queue with real-time Firestore synchronization, meaning administrators see new pending services appear immediately without needing to refresh the page. The queue includes search functionality by service title and provider name, and each listing displays a colour-coded SLA indicator showing how long the service has been waiting for review: green for less than 12 hours, orange for 12 to 48 hours, and red for over 48 hours, providing administrators with a priority-based workspace. When an administrator approves a service, the `notifyOnServiceApproval` Cloud Function detects the status change via an `onUpdate` Firestore trigger and immediately creates a notification document in the `notifications` collection containing the service title and ID, which in turn triggers the `sendPushOnNotificationCreate` function to deliver an FCM push notification to the provider's mobile device, completing the approval feedback loop in real time.

The ratings and reviews system was implemented through the `reviews` Firestore collection, with Firestore security rules that enforce the critical business constraint that only the seeker who participated in a booking that has reached the `completed` status may create a review for that booking. The rules validate the `bookingId`, `serviceId`, `providerId`, and `reviewerId` fields against the corresponding booking document and verify that the rating value falls within the 1–5 integer range. Reviews are immutable once submitted — the security rules do not permit update or delete operations — ensuring the integrity of the review record. Server-side rating aggregation was implemented through the `updateProviderRatingOnReviewCreate` Cloud Function, which performs a Firestore atomic transaction on the provider's user document to fetch the current `reviewCount` and `averageRating`, compute the new weighted average incorporating the new rating, and write both updated values back in a single all-or-nothing operation, preventing race conditions that could corrupt the aggregate in scenarios with concurrent review submissions.

### 4.3 Achievement of Objective 3 – Structured Booking and Real-Time Communication System

Objective 3 was fully achieved through the implementation of a formally defined booking lifecycle management system and a real-time in-app chat infrastructure, both of which are operational in the production environment and have replaced the informal, unstructured communication and coordination processes that motivated this objective.

The booking lifecycle is managed through the `bookings` Firestore collection, where each document stores the `serviceId`, `providerId`, `seekerId`, `amount`, `status`, `paymentStatus`, `reviewed` flag, and creation and update timestamps. The state machine defines the following valid transitions: a seeker creates a booking with `status: 'pending'`; the provider can transition it to `status: 'accepted'` or `status: 'rejected'`; from `'accepted'`, the provider can move it to `status: 'completed'`, or the seeker can move it to `status: 'cancelled'`; and a pending booking can also be cancelled by the seeker. These transitions are enforced at two layers simultaneously. The Flutter application enforces them at the UI layer by displaying only the contextually appropriate action buttons based on the current booking status and the current user's role. More importantly, the Firestore security rules enforce them at the database layer on every update operation, validating that the incoming status value represents a permitted transition from the current status and that the requesting user has the correct role to perform that transition. This dual-layer enforcement means that even if a malicious actor were to craft a direct Firestore write request outside the application, any invalid state transition would be rejected by the security rules.

The booking management screen provides a tab-based interface with tabs for All, Pending, Accepted, Completed, Cancelled, and Rejected statuses, allowing both seekers and providers to easily navigate their booking history and take the appropriate action on each booking. Providers use this screen to accept or reject incoming requests and to mark services as completed upon delivery. Seekers use it to monitor the status of their bookings, cancel pending ones if no longer needed, and initiate the review process once a booking reaches the completed state. Additionally, a request-based booking flow was implemented through the `requests` collection, enabling seekers to proactively post service requests directed at specific providers, who can then respond by creating an accepted booking, adding a bidirectional initiation model to the platform.

The real-time in-app chat system was built on the `messages` Firestore collection, where each message document is linked to a specific booking through a `chatId` field that stores the corresponding `bookingId`. The chat screen uses a Firestore real-time listener (`snapshots()`) to subscribe to the stream of messages for the current booking, rendering new messages instantly as they are written to the database without any manual refresh. The Flutter `StreamBuilder` widget bound to this listener rebuilds the message list automatically on each database event, providing a fluid, instant messaging experience. Firestore security rules restrict message creation and reading strictly to users whose UID matches either the `seekerId` or `providerId` of the associated booking, preventing any third-party access to booking conversations. The chat screen header displays the name of the other participant and the associated service title, providing clear context throughout the conversation.

### 4.4 Achievement of Objective 4 – Secure Digital Payment Processing

Objective 4 was fully achieved through the implementation of a multi-method payment system integrated with PayHere, Twilio, and SendGrid, covering the complete payment lifecycle from initiation through confirmation and receipt delivery, all of which is deployed and operational in the production environment.

The payment system was designed around three distinct payment pathways to accommodate the range of payment preferences common in the Sri Lankan market. The first pathway — credit and debit card payments — routes through the PayHere payment gateway, which is a licensed Sri Lankan payment processor supporting LKR-denominated transactions on Visa and MasterCard. The flow begins with the `createPayHereCheckoutSession` callable Cloud Function, which the mobile application invokes with the booking ID, amount, and order details. The function generates a PayHere checkout session with a unique order ID, configures a return URL and notify URL pointing to the deployed `payHereWebhook` Cloud Function endpoint, and returns the session parameters to the client for redirect. Upon payment completion, the PayHere gateway sends an HTTP POST callback to the `payHereWebhook` endpoint, which verifies the callback's authenticity by recomputing and comparing the PayHere-specified MD5 hash signature against the submitted values — ensuring the callback genuinely originated from PayHere — and then updates the payment document's status from `'pending_gateway'` to `'success'` or `'failed'` accordingly.

The second payment pathway — direct bank transfer — allows seekers who prefer interbank transfers to submit their transfer reference number and bank details through the `submitBankTransfer` callable Cloud Function, which records the submission and sets the payment status to `'pending_verification'`. An administrator then reviews the bank transfer details through the web administration dashboard and invokes the `verifyBankTransfer` callable Cloud Function to mark the payment as `'paid'` once the transfer is confirmed in the bank's records. This pathway is particularly important for the Sri Lankan market, where a significant segment of the adult population prefers bank transfers over card payments.

The third pathway — saved card tokenization — allows returning users to store a tokenized reference to their payment card without retaining the actual card number on the platform, enabling one-tap payment completion on future bookings. The `savedPaymentMethods` Firestore collection stores the token reference, card brand, last four digits, and expiry details, while the actual card data is retained exclusively by the PayHere gateway's token vault.

The payment status field follows a formally defined state machine: `'initiated'` → `'pending_gateway'` → `'success'` or `'failed'` for card payments, and `'initiated'` → `'pending_verification'` → `'paid'` for bank transfers. These transitions are validated by Firestore security rules, ensuring no client can write an arbitrary status value. The offer and discount system, implemented through the `offers` collection and the `OfferService` utility, checks for applicable discounts at checkout time — supporting flat-amount and percentage-based reductions targeted to a specific service, provider, or service category, with configurable minimum amount thresholds and time-bound activation windows — and applies the discount automatically to the displayed checkout amount.

Upon successful payment, the `dispatchPaymentReceipts` callable Cloud Function is invoked to deliver formal receipts through two channels: an SMS sent via the Twilio API to the seeker's registered phone number, and an email sent via the SendGrid API to the seeker's registered email address. Each receipt contains the booking reference, service details, amount paid, payment method, and transaction timestamp, providing complete transaction traceability and a professional record of the transaction for both parties.

### 4.5 Achievement of Objective 5 – Administrative Control and Monitoring Dashboard

Objective 5 was fully achieved through the development and deployment of a dedicated web-based administration panel built using Flutter Web, enabling the entire mobile and web application to be maintained as a single Dart codebase. The web application is deployed on Firebase Hosting and is live in the production environment. Access is strictly controlled — upon loading, the application verifies that the authenticated user's document in the `users` Firestore collection contains `role: 'admin'`, and immediately redirects any non-admin user without revealing any dashboard content.

The service moderation capability is the most operationally critical feature of the admin dashboard. The moderation screen presents all service listings organized into three tabs — Pending, Approved, and Rejected — with a real-time Firestore subscription ensuring that newly submitted services appear in the Pending tab instantaneously without manual refresh. Each listed service displays the service title, category, provider name, price, and the elapsed time since submission, with the elapsed time rendered as a colour-coded SLA indicator (green for under 12 hours, amber for 12–48 hours, red for over 48 hours) that allows administrators to prioritize long-waiting submissions and ensure timely review, which directly addresses the gap in quality governance that was identified in Problem 5. Approve and reject actions are available directly from the list, with the status update reflected immediately across all connected clients due to Firestore's real-time synchronization architecture.

The banner management module enables administrators to create and manage promotional banners that appear in a carousel on the mobile application's home screen. Each banner records a title, image URL, call-to-action link, active/inactive status, and optional role-based targeting. The promotion management module supports the creation and lifecycle management of broader marketing campaigns, with configurable start and end dates and descriptive content. The offers module provides granular discount creation, allowing administrators to specify whether a discount is a flat LKR amount or a percentage of the booking price, target the offer at a specific service, provider, or category, set a minimum booking amount threshold, and define the exact activation window, giving the platform powerful commercial tools for driving engagement and reward behaviour.

The analytics dashboard aggregates and visualizes platform-wide metrics — total user count, total bookings, booking status distribution, revenue totals, and provider performance — using the fl_chart library to render interactive bar charts and line graphs. User management capabilities allow administrators to inspect any user's profile details and modify their role if necessary, such as upgrading a trusted user to an administrator or downgrading a misclassified account. Together, these capabilities provide the governance infrastructure for the Lanka Connect platform to be operated as a well-managed, quality-controlled marketplace.

### 4.6 Achievement of Objective 6 – Secure, Scalable, and Real-Time Back-End Architecture

Objective 6 was fully achieved through the deployment of a comprehensive serverless back-end architecture on the Firebase platform, covering all six service pillars required by the objective: real-time data storage, secure identity management, media hosting, server-side business logic, push notifications, and role-based access control. The architecture is fully operational in production.

Cloud Firestore serves as the primary database, storing data across thirteen collections: `users`, `services`, `bookings`, `requests`, `reviews`, `messages`, `notifications`, `payments`, `paymentReceipts`, `providerBankAccounts`, `savedPaymentMethods`, `banners`, `promotions`, and `offers`. Firestore's real-time listener architecture means that every connected client — mobile or web — receives data updates instantaneously whenever a document they are subscribed to changes, without any polling. This enables the live booking status updates, instant chat messages, real-time notification delivery, and live service list refreshes that are central to the platform's user experience.

Firebase Authentication manages all user identity operations, including account creation, email/password login, session management, and password recovery via email link. Authentication tokens are automatically managed by the Firebase SDKs, securely stored on device, and included in every Firestore and Cloud Storage request, where the security rules validate the `request.auth.uid` on every operation.

Firebase Cloud Storage hosts all binary assets — service listing images uploaded by providers, user profile photographs, and promotional banner images — under a path-based access control scheme. Security rules restrict image uploads to 5 megabytes maximum file size and PNG/JPEG media types, prevent users from writing to any path other than their own designated directories, and enforce public read access only for approved asset paths.

Ten Cloud Functions deployed on the Node.js 22 runtime under Firebase Cloud Functions v2 handle all event-driven and callable server-side logic. These functions are deployed with a `maxInstances: 10` configuration that allows the platform to scale horizontally during peak usage while capping execution costs. The functions handle the complete range of server-side responsibilities: the service status guardrail, provider rating aggregation, approval notifications, push notification dispatch, payment session creation, payment callback verification, bank transfer submission and verification, receipt dispatch, and demo data seeding. Because these functions execute in an isolated server environment using the Firebase Admin SDK, they operate with privileges that transcend the client-facing security rules, enabling them to perform operations — such as forcibly setting a service's status on creation — that the client is expressly prohibited from performing directly.

Firebase Cloud Messaging delivers push notifications for all platform events, including booking status changes (accepted, rejected, completed), service listing approval or rejection, payment confirmations, and receipt deliveries. The `sendPushOnNotificationCreate` Cloud Function is triggered whenever a new notification document is created in the `notifications` collection, retrieves the recipient's registered FCM device token from their user document, and dispatches the notification with a structured payload that includes a notification type identifier and a reference ID for deep linking. The mobile application evaluates the deep link on notification tap and navigates directly to the relevant booking, service, or notification detail screen.

The role-based access control system was implemented entirely through Firestore security rules spanning all thirteen collections. The rules follow a consistent pattern: they first verify that the requesting user is authenticated, then resolve the user's role from the `users` collection, and then apply collection-specific permission logic. For example, the services collection rules verify that providers can only create listings with `status: 'pending'`, and can only update their own listings without modifying the status field; the bookings rules validate state machine transitions and cross-document consistency (checking that the referenced service exists and belongs to the referenced provider); the reviews rules verify booking completion and seeker identity; and the payments rules validate allowed status values and gateway identifier values. The security rules are deployed as part of the Firebase stack alongside the application code and execute server-side on Google's infrastructure, making them tamper-proof and providing an authoritative security boundary regardless of client behaviour.

The multi-environment deployment strategy separates the production Firebase project from a staging project and a local emulator configuration, each with its own Firestore database, authentication environment, storage bucket, and function deployments. The active environment is selected at compile time through the `APP_ENV` build flag, which routes Firebase initialization to the appropriate project-specific credentials file. This ensures that development and testing activities on staging or emulator environments have zero interaction with the production database or user data.

## 5. Future Recommendations

While all six objectives defined for the Lanka Connect project have been successfully achieved in the current production release, there are several areas where the platform can be meaningfully enhanced in future development cycles to increase its market reach, commercial viability, user experience quality, and operational robustness. The following recommendations are proposed based on the current system's limitations, emerging user needs, and broader industry trends in digital service marketplaces.

**Recommendation 1 – In-App Wallet and Escrow Payment System:**

The current payment system relies on direct gateway processing (PayHere) and bank transfers, both of which involve external parties or manual verification steps that introduce friction and delays into the payment process. A future development cycle should introduce an in-app digital wallet that allows both seekers and providers to maintain a balance within the platform, funded through card top-ups or bank transfers. More significantly, an escrow mechanism should be integrated into the booking workflow, whereby the seeker's payment is held in escrow at the time of booking acceptance and released to the provider only once the seeker confirms service completion. This would address the currently underserved dispute resolution needs of the platform, provide financial security for both parties, and reduce the incidence of non-payment disputes. Technically, this could be implemented through a dedicated escrow management service within Cloud Functions, with balance records maintained in Firestore and authoritative transaction logs stored for audit purposes.

**Recommendation 2 – Provider Identity Verification and KYC Integration:**

The current platform allows any user who registers as a provider to create service listings subject only to administrative content review. There is no verification of the provider's real-world identity, professional qualifications, or background. A future development priority should be the integration of a Know Your Customer (KYC) verification workflow that requires providers to submit a photograph of their National Identity Card (NIC), a selfie for liveness verification, and where applicable, professional certification documents (e.g., trade licences, academic qualifications for tutors). This data could be processed through a third-party identity verification API, and providers who complete KYC could receive a "Verified Provider" badge displayed prominently on their service listings and profile pages. This enhancement would substantially increase seeker confidence and differentiate high-quality providers from unverified ones, directly addressing the underlying trust deficit identified in Problem 2.

**Recommendation 3 – Machine Learning-Based Service Recommendations:**

The current service discovery experience is entirely driven by the seeker's explicit filter selections and keyword searches, with no personalized recommendations based on the seeker's individual history or preferences. As the platform accumulates data — booking history, search patterns, category affinities, location data, and rating interactions — it would become viable to develop a recommendation engine that surfaces relevant service listings that a particular seeker has not explicitly searched for but is statistically likely to find valuable. This could be implemented through a collaborative filtering model or a content-based recommendation model deployed as a Cloud Function that precomputes recommendation sets for active users on a scheduled basis and stores them in Firestore for low-latency retrieval. In the near term, simpler heuristic-based recommendations (e.g., "Services popular in your district" or "Top-rated services in your most-booked category") could be implemented with minimal infrastructure and provide meaningful value while sufficient data accumulates for a machine learning model.

**Recommendation 4 – Multi-Language Support (Sinhala, Tamil, and English):**

Lanka Connect currently operates entirely in English, which presents a significant accessibility barrier for a large segment of Sri Lanka's population who are more comfortable transacting in Sinhala or Tamil. Both Sinhala and Tamil are official languages of Sri Lanka, and for many service providers operating in rural or non-urban areas, English may not be their primary language, limiting both their ability to create effective service listings and their confidence in using the platform. The Flutter framework's internationalization (i18n) infrastructure supports runtime locale switching, and a future development cycle should implement full string externalization across all screens, translate all UI strings into Sinhala and Tamil by professional language specialists, and allow users to select their preferred display language during onboarding or from the settings screen. This would dramatically expand the serviceable user base and position Lanka Connect as a genuinely inclusive platform for all Sri Lankans.

**Recommendation 5 – In-App Dispute Resolution and Mediation System:**

The current platform does not provide a structured mechanism for resolving disagreements between seekers and providers after a booking has been completed or disputed. A formal dispute resolution system should be developed that allows either party to raise a dispute against a booking, submit evidence (photos of completed or incomplete work, chat transcripts, payment receipts), and request platform-mediated resolution. The dispute would enter an administrative review queue in the admin dashboard, where an administrator could review the evidence, communicate with both parties through the platform, and render a resolution decision — such as approving a refund, releasing escrow, or taking action against a user account. Automated escalation timelines and resolution tracking would prevent disputes from being left unaddressed. This feature would significantly improve platform trustworthiness and give seekers and providers a safe, structured recourse when disagreements arise.

**Recommendation 6 – Provider Subscription and Tier-Based Listing Plans:**

The current platform does not have a defined revenue model for the service marketplace operator; all features are available to providers at no cost. A sustainable future monetization strategy should include the introduction of tiered subscription plans for service providers. A Basic tier (free) would allow a limited number of service listings; a Professional tier (monthly fee) would provide enhanced listing visibility, priority placement in search results, promotional banner eligibility, and access to the advanced analytics dashboard; and a Premium tier would offer featured placement on the home screen, verified badge priority, and extended promotional reach. This revenue model is well-established among international service marketplace platforms and would enable Lanka Connect to operate as a self-sustaining commercial platform without relying entirely on advertising or transaction fees.

**Recommendation 7 – Enhanced In-App Chat with Media Sharing:**

The current chat system supports text messages only. In practice, effective communication about service requirements frequently involves the exchange of photographs — a seeker might want to share a photo of a leaking pipe to help the plumber understand the job scope, or a provider might want to share a progress photo during service execution. Extending the chat system to support image attachments (uploaded to Firebase Cloud Storage with size and format restrictions), short video clips, and potentially document attachments would substantially improve the utility of the in-app communication channel. This enhancement would reduce the need for seekers and providers to exchange personal phone numbers or move conversations to external messaging applications, keeping all booking-related communication within the platform and maintaining the complete communication record in a structured, searchable format.

**Recommendation 8 – Advanced Analytics and Reporting for Providers:**

The current provider dashboard presents summary metrics including total bookings, revenue, and average rating. While useful for a high-level performance overview, professional service providers managing significant booking volumes would benefit from a more granular analytics suite. Future enhancements should include: time-series charts showing booking volume and revenue trends over selectable periods (weekly, monthly, quarterly); booking conversion funnel analysis showing how many bookings progress from pending to accepted to completed; customer retention metrics tracking repeat seekers; category and location demand heatmaps for strategic service offering decisions; and comparison metrics against anonymized platform averages. This richer data would empower providers to make more informed business decisions, optimize their pricing, improve their service quality, and grow their revenue through the platform.

**Recommendation 9 – Integration of Additional Payment Gateways and Digital Wallets:**

While PayHere and bank transfers cover the most common payment methods in the current Sri Lankan digital payments landscape, the market is evolving rapidly. Mobile wallet services such as FriMi and eZ Cash have growing user bases, and QR-code-based payments via LankaPay are gaining adoption. Future payment gateway integrations should expand to include these popular local digital payment options, as well as integration with the national LankaPay QR standard for in-app payment. For providers who are international or seek to serve expatriate clients, support for international payment processors such as Stripe could also be explored. Expanding the payment options reduces friction for users whose preferred payment method is not currently supported and increases the platform's transaction completion rate.

**Recommendation 10 – Progressive Web App (PWA) Capability for Service Seekers and Providers:**

The current web application is restricted to the admin dashboard, and service seekers and providers must install the mobile application to access the platform. Converting the web application into a full Progressive Web App (PWA) with offline capability through service worker caching, an installable web app manifest, and a responsive design optimized for both desktop and mobile browsers would make the platform accessible to users who cannot or prefer not to install a native application. This is particularly relevant for service seekers who want to book a service quickly from their desktop browser at work, or for providers who manage their listings from a desktop computer. Flutter Web already supports PWA features including manifest configuration and service worker generation, and a relatively modest development investment would yield a significantly broader reach for the platform.

## 6. Summary of Conclusion

Lanka Connect was conceived, designed, and delivered as a comprehensive cross-platform service marketplace to directly address five significant structural problems in the Sri Lankan local service industry: the absence of a centralized digital discovery platform, the lack of trust and quality assurance mechanisms, the inefficiency of informal booking and communication processes, the insecurity and limitation of cash-only payment practices, and the absence of any administrative oversight governing service quality. Six corresponding objectives were defined prior to development, and the analysis of the completed production system confirms that all six have been successfully achieved. The multi-category service marketplace was implemented with keyword search, multi-parameter filtering, Google Maps geolocation discovery, and interactive service maps across nine service categories on the mobile application. The trust and quality assurance framework was delivered through an immutable rating and review system with atomic server-side rating aggregation and a server-enforced service moderation pipeline with SLA tracking on the web admin dashboard. The structured booking lifecycle, enforced simultaneously at the application and database layers through Firestore security rules, and the real-time in-app chat system together replaced the informal coordination processes that previously led to disputes and communication breakdowns. The multi-method digital payment system — integrating PayHere for card processing, bank transfer verification with administrative approval, saved card tokenization, discount offer application, and automated SMS and email receipt delivery through Twilio and SendGrid — established a complete, traceable, and secure financial transaction infrastructure. The web-based administration dashboard deployed on Firebase Hosting provides comprehensive governance tools covering service moderation, banner and promotion management, analytics, and user administration. The entire platform is backed by a serverless Firebase architecture, with ten deployed Cloud Functions and comprehensive role-based Firestore security rules across thirteen data collections and four user roles, operating under a multi-environment deployment model that supports continued development without production impact. Lanka Connect is live in production, and the platform's architecture and feature set create a strong foundation upon which the recommended enhancements — including in-app escrow payments, provider KYC verification, machine learning recommendations, multi-language support, dispute resolution, and provider subscription tiers — can be progressively built to evolve Lanka Connect into a fully mature, commercially self-sustaining service marketplace for Sri Lanka.

## 8. Non-Functional Requirements

Non-functional requirements define the qualities, constraints, and operational standards that the Lanka Connect system must satisfy beyond simply performing its explicitly required functions. While functional requirements specify what the system does, non-functional requirements specify how well it does it and under what conditions it must remain correct and safe. For a production-deployed service marketplace handling real financial transactions, real identity data, and real-time communication between users who have no prior relationship with each other, non-functional qualities such as security, data integrity, performance, and reliability are not optional enhancements — they are foundational to the trustworthiness and commercial viability of the platform.

The non-functional requirements for Lanka Connect were identified through an analysis of the system's operational environment and the specific risks inherent in its architecture. The platform is deployed to the internet, accepting connections from anonymous users on mobile devices and web browsers, and it processes sensitive data including personal contact information, bank account details, and payment transaction records. It must operate correctly across a wide range of device hardware and operating system versions, and must maintain data consistency under concurrent writes from geographically distributed users. It must recover gracefully from partial failures and network interruptions, and it must provide administrators with sufficient visibility into system health to detect and address issues before they impact users.

The following non-functional requirements were identified, categorized, and addressed in the Lanka Connect implementation. Each requirement has been assigned to one of seven categories — Performance, Concurrency, Compatibility, Security, Data Integrity, Scalability, Usability, Maintainability, Reliability, and Localization — to facilitate systematic review and validation:

The security category of non-functional requirements deserves particular elaboration given their critical importance in a platform that handles real financial transactions. The Firestore security rules are the primary enforcement layer, and they are designed using a defence-in-depth principle: even if an attacker bypasses the mobile application's UI layer entirely and sends raw Firestore API requests, the security rules will independently evaluate every request against the authentication state, the user's registered role, and the business logic constraints embedded in the rule conditions. The payment webhook NFR (NFR-07) is implemented through a server-side Cloud Function that reconstructs the expected signature from the shared secret and compares it with the received signature using a constant-time comparison, preventing timing-based attacks. The bank account encryption requirement (NFR-08) ensures that even if the Firestore database were exposed through a misconfiguration, raw account numbers would not be directly accessible.

The data integrity NFR (NFR-09) represents a particularly important architectural decision: by encoding allowed state transitions directly in the Firestore security rules using conditional logic on the `request.resource.data` and `resource.data` fields, the platform guarantees that no invalid booking or payment status transition can ever be stored, regardless of what code or actor attempts the write. This provides a stronger guarantee than application-level validation alone, which could be bypassed by a compromised client. The scalability requirement (NFR-10) reflects the decision to use Cloud Functions v2 with explicit concurrency controls, which provides automatic horizontal scaling while preventing unbounded invocation storms from consuming platform quotas.

## 9. Implementation Progress

The Lanka Connect platform has been developed from initial requirements analysis through to full production deployment. This section documents the current implementation state of each major system component, providing evidence that all planned features have been built, tested, and are operational in the live environment. The implementation spans three distinct tiers: the Flutter-based mobile application for Android and iOS, the Flutter Web administration dashboard deployed to Firebase Hosting, and a Node.js TypeScript Cloud Functions back end deployed on Firebase Cloud Functions v2. All components are connected to a shared Cloud Firestore database and operate under a unified Firebase Authentication layer.

The development approach followed an iterative construction model in which each functional domain — authentication, service management, booking, payments, communication, and administration — was implemented vertically through all tiers in sequence, allowing end-to-end testing of each domain before proceeding to the next. This approach minimized integration risk and provided early, testable builds at each milestone. The multi-environment configuration (production, staging, local emulator) enabled parallel testing without risking contamination of live production data.

### 9.2 Mobile Application – Implemented Features

The mobile application is the primary user-facing component of the Lanka Connect platform and the release artefact that end consumers — both seekers and providers — interact with on their Android and iOS devices. It was developed entirely in Flutter using the Dart programming language, following the provider-based state management architecture, and organized into a modular screen and widget hierarchy that maps directly to the product's functional domains. The single-codebase approach allowed simultaneous delivery of fully functional Android and iOS builds without platform-specific reimplementation of any core logic.

The application implements a multi-entry-point routing strategy that directs users to the appropriate initial screen based on their authentication state and registered role on launch, ensuring that seekers, providers, admins, and guests each enter through an experience tailored to their capabilities. All navigation transitions are handled through Flutter's named routing system, and all data access is mediated through dedicated service classes that abstract the Firestore read/write operations, keeping screen-level code free of direct database logic and making the application testable at the service layer.

**Authentication Module:**
- Email/password registration with role selection (Seeker/Provider), backed by Firebase Authentication `createUserWithEmailAndPassword` and a corresponding user document written to the `users` Firestore collection on first sign-in
- Secure login with session persistence through Firebase Authentication's built-in token refresh, ensuring that users do not need to re-authenticate after an app restart unless they explicitly log out
- Password reset via email link initiated through Firebase Authentication's `sendPasswordResetEmail`, providing a self-service recovery path without any bespoke back-end implementation
- Session state management using a stream of `User?` objects from Firebase Authentication's `authStateChanges`, which drives the root navigation decision at the application entry point

**Service Discovery Module:**
- Browse all Firestore-approved services with dynamically rendered service cards showing the service thumbnail, title, provider name, aggregate rating with star representation, review count, price, and category badge
- Full-text keyword search implemented through client-side filtering on title and description fields, returning results from the approved services snapshot in real time as the user types
- Category, location (district and city), and price range filters implemented as composable client-side predicates applied to the live Firestore snapshot, allowing multiple filters to be active simultaneously
- "Near Me" geolocation-based discovery using the Geolocator package to access the device's GPS, with the Haversine formula applied to compute the great-circle distance between the device's coordinates and each service's stored GPS coordinates, filtering to only services within the user-configurable radius of 10 to 100 kilometres
- Interactive Google Maps view rendered through the Google Maps Flutter plugin displaying approved services as custom markers at their respective GPS coordinates, with tap-to-preview overlays linking to the full detail screen
- Full service detail screen with a swipeable image gallery rendered from Firebase Storage URLs, a formatted description block, provider summary card, aggregate rating display with review count, recently submitted reviews list, and an embedded map tile showing the service location

**Service Management Module (Provider):**
- Multi-step service creation form collecting title, category (from nine predefined options), price, descriptive text, and GPS coordinates (captured by pressing a "Capture My Location" button that invokes Geolocator), followed by a multi-image picker that uploads selected images to Firebase Cloud Storage and stores the resulting download URLs in the service document
- Service status dashboard showing the provider's own services organized by moderation status (Pending, Approved, Rejected), providing immediate feedback on the moderation outcome of each submitted listing
- Edit and delete actions on own service listings with Firestore document updates and Cloud Storage image deletion for removed images

**Booking Module:**
- One-tap booking creation from the service detail screen, which creates a Firestore document in the `bookings` collection with the seeker's UID, provider's UID, service reference, amount, and initial status of `pending`
- Booking management screen with a TabBar-based status filter (All, Pending, Accepted, Completed, Cancelled, Rejected) and real-time status updates as providers respond to requests
- Provider-side action buttons on booking detail screens to accept or reject pending bookings, and to mark accepted bookings as completed, with all status transitions validated by Firestore security rules before the write is committed
- Service request flow where seekers can direct specific requests to a provider via the `requests` collection, and providers can respond with an accepted booking creation, enabling provider-initiated engagement

**Chat Module:**
- Real-time messaging powered by Firestore `onSnapshot` listeners on the `messages` sub-collection of each booking document, delivering new messages to both participants within milliseconds without polling
- Composer row with text input, message validation (non-empty check), and send button creating a new Firestore message document with sender UID, timestamp, and content
- Firestore security rules restricting read and write access to the chat sub-collection exclusively to the `seekerId` and `providerId` fields stored in the parent booking document, enforcing participant-only access at the database level

**Payment Module:**
- PayHere credit/debit card checkout initiated by calling the `createPayHereCheckoutSession` Cloud Function, which returns a signed checkout URL that is opened in the device's browser, with the payment result communicated back to the app via a deep link redirect and confirmed by the `payHereWebhook` Cloud Function
- Bank transfer submission screen where seekers enter their transfer reference number, triggering the `submitBankTransfer` Cloud Function to record the pending verification entry in the `payments` collection
- Saved card management screen listing previously tokenized cards stored in the `savedPaymentMethods` collection, allowing seekers to select a saved card at checkout
- Discount application logic that queries the `offers` collection for an applicable offer at the time of payment initiation and reduces the displayed checkout amount by the applicable flat or percentage discount before submitting the payment session request

**Review Module:**
- Star rating widget (1–5 stars, half-star precision) and optional textual comment field, accessible only from the booking detail screen of a booking in `completed` status, preventing premature review submission
- Review submission writing to the `reviews` collection and triggering the `updateProviderRatingOnReviewCreate` Cloud Function, which executes an atomic Firestore transaction to update the provider's `averageRating` and `reviewCount` fields
- Review display on both the service detail screen (showing recent reviews with reviewer name, date, rating, and comment text) and the provider profile screen (showing aggregate rating prominently)

**Notification Module:**
- Push notification delivery via Firebase Cloud Messaging, with the device FCM token stored in the user's Firestore profile document and refreshed on each login to ensure up-to-date delivery targets
- Background and foreground notification handling using the `firebase_messaging` package with a distinguishing banner display for foreground-received notifications
- Notification centre screen showing all notification documents from the `notifications` collection for the authenticated user, with unread notifications visually highlighted and a "mark all as read" batch-update action
- Deep link payload in each notification's FCM message data field routing the user directly to the relevant screen (booking detail, service detail, payment detail) when the notification is tapped

**Profile and Provider Dashboard Module:**
- Profile editing screen covering name, contact number, district, city, biography/skills (for providers), and profile photo with image picker integration and Firebase Cloud Storage upload
- Provider dashboard presenting a monthly booking summary, cumulative revenue figure, average rating, review count, level indicator based on completed booking milestones, a recent bookings list, and performance trend charts rendered using the fl_chart library

### 9.3 Web Application – Implemented Features

The web administration dashboard is a dedicated Flutter Web application compiled from a separate entry point that is configured to recognize only the admin role and to redirect unauthorized users immediately. It is deployed to Firebase Hosting and provides platform administrators with a structured operational interface for all moderation, content management, and analytical tasks. The web application shares the same Firebase back-end as the mobile application and operates on the same Firestore collections, ensuring that changes made through the web panel are immediately reflected in the mobile application's live data stream and vice versa.

The web application was designed with a focus on operational efficiency for administrative workflows. The primary navigation pattern uses a persistent sidebar that provides labelled links to each administrative module, with the main content area rendering the selected module's full list or detail view. All list views implement real-time Firestore listeners so that new submissions or incoming data changes are reflected without requiring a page refresh, which is particularly important for the service moderation queue where new listings may arrive while an administrator has the queue open.

**Service Moderation Dashboard:**

The service moderation module is the highest-priority administrative interface in the web application, responsible for reviewing and acting on every newly submitted service listing before it becomes visible to seekers. The moderation queue displays all services in `pending` status with their title, category, price, submitting provider's name, and creation timestamp. Each listing card includes a thumbnail preview of the uploaded images and a one-click approve or reject action. The SLA (Service Level Agreement) tracking system uses colour-coded visual indicators based on time elapsed since submission: green for listings awaiting review for fewer than 12 hours, amber for 12 to 48 hours, and red for listings that have been waiting more than 48 hours, providing immediate visual prioritization of the queue. When an administrator approves or rejects a listing, the `notifyOnServiceApproval` Cloud Function fires automatically to create a push notification document for the listing's provider, informing them of the moderation outcome.

**Banner, Promotion, and Offer Management:**

The content management modules — Banner, Promotion, and Offer Management — provide administrators with the tools to configure the marketing and commercial incentive content displayed to mobile users. The Banner Management module allows administrators to create promotional banners that appear in the carousel on the mobile application's home screen, uploading images to Firebase Cloud Storage, configuring a call-to-action URL, and toggling each banner between active and inactive states. The Promotion Management module supports the creation of broader marketing campaigns with descriptive content and time-bound active periods. The Offer and Discount Management module is the most commercially sensitive of the three: administrators can create targeted discounts specifying whether the reduction is a flat LKR amount or a percentage, the minimum booking amount required to qualify, the target scope (a specific service by ID, a specific provider by UID, or an entire category), and the activation start and end dates. Mobile users encounter these discounts transparently at the payment checkout stage.

**Analytics Dashboard and User Management:**

The analytics dashboard aggregates Firestore data to present platform operators with a high-level view of system health and commercial performance, displaying total registered user count, total bookings segmented by status, cumulative payment revenue, and service volume trends over time using interactive chart visualizations rendered by the fl_chart library. The User Management module allows administrators to look up registered user accounts, view their full profile details, and modify their role assignment, which is the mechanism through which new administrators are provisioned on the platform.

### 9.4 Cloud Functions – Implemented Backend Logic

The Cloud Functions layer forms the critical server-side enforcement and integration tier of the Lanka Connect platform. Ten functions were designed and deployed, each addressing a specific need that cannot be safely or reliably handled in client-side code — either because the operation requires privileged access, must not be bypassable by a malicious client, or involves integration with third-party APIs whose credentials must remain server-side secrets.

All Cloud Functions were written in TypeScript, compiled to JavaScript for the Node.js 22 runtime, and deployed to Firebase Cloud Functions v2 with an explicit `maxInstances: 10` concurrency cap to provide controlled horizontal scaling. The TypeScript codebase is organized in the `functions/src/` directory with modular files per functional domain, and the build is managed through the `tsconfig.json` configuration targeting ES2017 module output.

The following Cloud Functions were implemented in TypeScript (Node.js 22 runtime) and deployed to Firebase Cloud Functions v2:

| Function | Type | Implementation Details |
|----------|------|----------------------|
| `setServicePendingOnCreate` | Firestore onCreate trigger | Automatically sets `status: 'pending'` on all new service documents, preventing provider self-approval. This function fires server-side before the document is visible to any client query, guaranteeing that no approved status can originate from a client write. |
| `notifyOnServiceApproval` | Firestore onUpdate trigger | Watches for `status` field changes on service documents; when the status transitions to `approved` or `rejected`, creates a notification document in the `notifications` collection for the owning provider, including the service title, new status, and a timestamp. |
| `updateProviderRatingOnReviewCreate` | Firestore onCreate trigger | On every new document in the `reviews` collection, executes an atomic Firestore transaction that reads the current `averageRating` and `reviewCount` from the provider's user document, computes the new weighted average incorporating the incoming review's star rating, and writes the updated values back — all within a single transaction to prevent race conditions under concurrent review submissions. |
| `sendPushOnNotificationCreate` | Firestore onCreate trigger | On every new notification document, reads the recipient user's FCM token from their Firestore profile, constructs an FCM message with a title, body, and data payload containing the notification type and relevant entity ID, and calls the Firebase Admin SDK's `messaging.send()`. Supports an admin broadcast channel (`__admins__`) by fanning out FCM messages to all users with the admin role when the recipientId is set to the broadcast channel identifier. |
| `createPayHereCheckoutSession` | Callable function | Accepts the booking ID, amount, seeker details, and service description; constructs a signed PayHere checkout session request using the merchant credentials stored in Firebase environment configuration; returns the checkout URL to the mobile application for browser-based card payment processing. |
| `payHereWebhook` | HTTP endpoint | Receives POST callbacks from the PayHere gateway upon payment completion or failure; reconstructs the expected MD5 signature from the order ID, amount, currency, and merchant secret; compares it with the received signature to reject spoofed callbacks; updates the corresponding payment document's status in Firestore on verified success. |
| `submitBankTransfer` | Callable function | Accepts the booking ID and bank transfer reference number from the caller; validates that the caller is the seeker associated with the booking; creates a payment document with `status: 'pending_verification'` and the transfer reference recorded, ready for admin review. |
| `verifyBankTransfer` | Callable function | Admin-only callable function (enforced by checking `request.auth.token.role === 'admin'` inside the function) that accepts a payment document ID and an approval decision; on approval, sets the payment status to `paid` and triggers the `dispatchPaymentReceipts` function invocation. |
| `dispatchPaymentReceipts` | Callable function | Integrates with the Twilio REST API to dispatch an SMS receipt to the seeker's registered phone number and with the SendGrid API to dispatch a formatted email receipt to the seeker's registered email address; called automatically on successful payment confirmation from both the card and bank transfer payment paths. |
| `seedDemoData` | Callable function | Admin-only function that creates a complete demonstration data set: a demo provider user, a set of sample service listings across multiple categories with pre-populated images, a completed booking, a five-star review, and a notification — enabling platform demonstrations without manual data entry. |
